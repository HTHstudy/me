# React, but with Reasons

🌐 [English](../en) | **한국어**

---

이것은 **내가 React와 프론트엔드 아키텍처를 어떻게 이해하는지**에 대한 개인적인 포트폴리오입니다.

튜토리얼이 아니며, 모범 사례 모음도 아닙니다.
대신, 다음을 기록합니다:

- 내가 프론트엔드 문제를 어떻게 바라보는지,
- 트레이드오프를 어떻게 평가하는지,
- 그리고 현실적인 제약 속에서 아키텍처 결정을 어떻게 내리는지.

특정 패턴이 _왜_ 작동하는지(또는 실패하는지)에 관심이 있다면,
여기가 좋은 출발점입니다.

---

## 이 레포지토리의 구성

### Mental Model (사고 모델)

코드를 작성하기 _전에_ 프론트엔드 시스템을 어떻게 개념화하는지.

이 섹션은 다음에 집중합니다:

- 실용적인 도구로서의 관심사의 분리,
- UI 로직 vs 도메인 로직,
- Headless UI와 경계 중심 설계,
- 핵심 설계 제약으로서의 접근성,
- 그리고 프론트엔드 시스템의 책임 경계.

→ **내가 어떻게 생각하는지** 이해하고 싶다면 여기서 시작하는 것을 추천합니다.

---

### Decisions over Patterns (패턴보다 결정)

패턴은 결과물이지, 시작점이 아닙니다.

이 섹션은 다음을 탐구합니다:

- 제어 컴포넌트 vs 비제어 컴포넌트,
- 상태 관리 결정 기준,
- React Context가 전역 상태가 아닌 이유,
- Context를 범위가 지정된 의존성 주입(DI)으로 사용하기,
- 그리고 주어진 문제에 적합한 React 원시 요소 선택하기.

→ **React 특화 설계 결정**에 관심이 있다면 추천합니다.

---

### Anti-patterns I've seen in the wild (현장에서 본 안티패턴들)

시간이 지나면서 나타나는 일반적인 실패 모드들.

이 섹션은 다음을 기록합니다:

- callback ref의 오용과 숨겨진 결합,
- 암시적 상태와 플래그 기반 로직,
- UI, 도메인, 인프라 간의 경계 누수,
- 그리고 StrictMode가 취약한 설계를 어떻게 드러내는지.

이것들은 "규칙"이 아니라, **비용이 이점을 초과하기 시작하는 시점**에 대한 관찰입니다.

---

### Case Studies (사례 연구)

익명화된 실제 리팩토링 스토리.

각 사례 연구는 다음에 집중합니다:

- 원래 설계가 무엇을 최적화했는지,
- 어디서부터 무너지기 시작했는지,
- 무엇이 변경되었는지,
- 그리고 리팩토링 후에도 어떤 트레이드오프가 남아있었는지.

(이 섹션은 시간이 지나면서 성장합니다.)

---

## 읽는 방법

- 큰 그림을 위해 **Mental Model**부터 시작하세요
- 구체적인 React 예제를 위해 **Decisions over Patterns**로 이동하세요
- 일반적인 함정을 이해하기 위해 **Anti-patterns**를 읽으세요
- 실용적이고 경험 중심의 인사이트를 위해 **Case Studies**를 활용하세요
