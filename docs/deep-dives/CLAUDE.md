# Deep Dives 문서 작성 규칙

## 목적

- 개념의 본질을 탐구하여 독자가 스스로 판단할 수 있게 돕는다.
- Decisions 문서에서 판단을 내리기 위해 필요한 개념적 기반을 제공한다.

## 다른 섹션과의 관계

- Mental Model: "왜 이런 일이 생기는가" → 힘과 긴장의 묘사
- Decisions over Patterns: "무엇을 선택할 것인가" → 결정 과정과 판단 기준
- Deep Dives: "이것은 무엇인가" → 개념의 본질 탐구
- Decisions 문서에서 개념 이해가 필요한 지점이 생기면 Deep Dives를 참조한다.

## 형식

- 문서 상단에 제목과 핵심 질문형 인용구를 둔다.
- 인용구는 "이 개념의 본질이 무엇인가"를 드러낸다.

## 필수 구성 요소

### 1. 개념 정의
- 이것은 무엇인가를 명확히 서술한다.
- 정의는 단정적이어도 된다.

### 2. 왜 존재하는가
- 이 개념이 해결하려는 문제를 설명한다.
- 이것이 없었을 때 어떤 어려움이 있었는지 보여줄 수 있다.

### 3. 사용 사례
- 코드 예시와 함께 구체적인 상황을 보여준다.
- 하나의 사례가 아니라 여러 맥락에서의 사용을 다룬다.

### 4. 정리하며
- 개념의 핵심을 다시 짚는다.
- 관련된 Decisions 문서로의 연결을 안내할 수 있다.

## 선택적 구성 요소

- **흔한 오해**: 자주 혼동되거나 잘못 사용되는 지점이 있는 경우
- **다른 개념과의 비교**: 유사한 개념과의 차이가 이해에 도움이 되는 경우

## 작성 관점

- "이것은 ~이다" (정의)와 "이것은 ~를 위해 존재한다" (목적)는 단정할 수 있다.
- 사용 사례는 "이런 상황에서 이렇게 쓰인다"로 보여주되, 유일한 방법으로 제시하지 않는다.
- 개념 설명에 필요한 만큼의 코드를 포함하되, 튜토리얼이 되지 않는다.

## 금지

- "이것을 ~에 써야 한다"는 표현을 피한다. (사용법 강제)
- "이것이 ~보다 낫다"고 단정하지 않는다. (비교 우위 단정)
- 개념 설명 없이 사용법만 나열하지 않는다.

## 위반 신호

- API 문서처럼 읽히면 위반이다. (무엇인가 없이 어떻게만 있는 경우)
- 특정 사용법이 유일한 정답처럼 서술되면 위반이다.
- "왜 존재하는가" 없이 "어떻게 쓰는가"만 있으면 위반이다.
