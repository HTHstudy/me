# Why, Then How 문서 작성 규칙

## 목적

- 실제 문제를 인식하고, 그 문제를 어떻게 해결했는지 보여준다.
- 앞선 섹션들의 사고를 거친 후 도출된 구체적인 구조를 기록한다.

## 다른 섹션과의 관계

- Mental Model: "왜 이런 일이 생기는가" → 힘과 긴장의 묘사
- Decisions over Patterns: "무엇을 선택할 것인가" → 결정 과정과 판단 기준
- Deep Dives: "이것은 무엇인가" → 개념의 본질 탐구
- Why, Then How: "어떤 문제를 인식하고 어떻게 해결했는가" → 적용 사례
- 앞선 섹션들이 사고의 과정을 다뤘다면, 이 섹션은 그 사고의 결과물을 다룬다.

## 형식

- 문서 상단에 제목과, 해결하려는 문제를 드러내는 인용구를 둔다.

## 필수 구성 요소

### 1. 문제 인식
- 어떤 상황에서 어떤 문제를 겪었는지 구체적으로 서술한다.
- 문제가 왜 문제인지를 설명한다.

### 2. 해결 구조
- 문제를 해결하기 위해 도출한 구조를 보여준다.
- 코드 예시와 디렉토리 구조 등 구체적인 형태를 포함한다.
- 각 구성 요소의 역할과 책임을 설명한다.

### 3. 왜 이 구조인가
- 이 구조가 앞서 인식한 문제를 어떻게 해결하는지 연결한다.
- 다른 접근이 아닌 이 구조를 선택한 이유를 설명한다.

### 4. 정리하며
- 핵심 구조를 요약한다.

## 작성 관점

- "이 문제를 이렇게 풀었다"로 서술한다.
- 구체적인 구조와 코드를 보여줘도 된다.
- 같은 문제에 대해 다른 해결이 가능함을 인정한다.

## 금지

- "이것이 유일한 정답이다"라고 단정하지 않는다.
- 문제 인식 없이 구조만 제시하지 않는다.

## 위반 신호

- 문제가 없이 구조만 나열되면 위반이다.
- "왜 이 구조인가"가 빠져 있으면 위반이다.
- 튜토리얼처럼 따라하기만 가능하고 판단의 근거가 없으면 위반이다.
