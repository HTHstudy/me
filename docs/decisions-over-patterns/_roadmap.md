# Decisions over Patterns 목차 로드맵

> 이 파일은 작성 계획을 위한 임시 파일입니다.  
> 문서를 작성하면서 업데이트됩니다.

---

## 문서 순서 (개발자의 사고 흐름)

컴포넌트가 복잡해졌을 때 개발자가 던지는 질문의 순서를 따릅니다.

```
컴포넌트가 복잡하네?
    ↓
불필요한 상태가 없나? → "이 값은 상태여야 하는가"
    ↓
상태의 위치는 여기가 맞을까? → "상태를 어디에 둘 것인가"
    ↓
상태 정리했는데 아직 복잡하네 → "이 컴포넌트는 나눠야 하는가"
    ↓
나눴으면 어떻게 연결하지? → "이 컴포넌트는 제어되어야 하는가"
    ↓
로직이 반복되네 → "이 로직은 훅이어야 하는가"
```

| 순서 | 문서 | 맥락 | 상태 |
|------|------|------|------|
| 1 | 이 값은 상태여야 하는가 | 불필요한 상태 제거 | ✅ 완성 |
| 2 | 상태를 어디에 둘 것인가 | 상태 위치 정리 | ✅ 완성 |
| 3 | 이 컴포넌트는 나눠야 하는가 | 구조 분리 | ✅ 완성 |
| 4 | 이 컴포넌트는 제어되어야 하는가 | 분리된 컴포넌트 설계 | ⏳ 예정 |
| 5 | 이 로직은 훅이어야 하는가 | 로직 재사용 | ⏳ 예정 |
| ... | (작성하면서 추가) | | |

---

## Deep Dives와의 관계

Decisions 문서에서 개념 이해가 필요한 지점이 생기면 Deep Dives를 참조합니다.

| Decisions 문서 | 참조할 Deep Dives |
|----------------|-------------------|
| 이 컴포넌트는 나눠야 하는가 | 합성 패턴 |
| 이 컴포넌트는 제어되어야 하는가 | 제어 vs 비제어 컴포넌트 |
| 상태를 어디에 둘 것인가 | React Context의 역할 |

---

## 메모

- 결정 문서는 "정답"을 주지 않는다
- 판단 과정과 트레이드오프를 묘사한다
- 독자가 자신의 상황에서 결정을 내릴 수 있게 돕는다
- 개념 이해가 필요하면 Deep Dives를 참조한다
